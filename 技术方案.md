# 一起算算 · 星座星盘系统 技术方案文档

## 一、项目概述

基于占星学的全栈 Web 应用，提供星盘计算、AI 占星师问答、每日运势、社区互动四大核心功能。采用前后端分离架构，后端提供 RESTful API，前端为纯原生多页面应用（MPA）。

| **跨会话记忆** | `user_memories` 表存储 AI 提取的实体/偏好/事实，每次对话后异步提取，注入 system prompt |

| 降级策略 | Agent 执行失败时自动降级为直接 LLM 调用（不带工具），保证响应可用性 |
| 工具调用 | AI 可主动调用两个工具：`search_web`（DuckDuckGo 实时搜索）、`query_user_data`（查询用户星盘/运势历史） |
| Agent 循环 | `call_ai_with_tools()` 实现最多 3 轮工具调用循环，AI 返回 `tool_calls` 时自动执行并将结果追加到 messages 继续对话 |
| 对话持久化 | 每轮问答存入 `chat_logs` 表，下次加载时取最近 30 条作为历史上下文 |
| AI 角色设定 | 将 AI 定位为专业占星师，根据用户星盘数据提供符合占星学理论的解读 |

### 2.3 占星算法

| 项目 | 说明 |
|------|------|
| **当前方案** | flatlib（Python 封装的 Swiss Ephemeris），专业占星学库，精确计算行星位置、宫位、上升点，支持逆行检测 |
| **降级方案** | 基于 `hashlib.md5(出生时间 + 城市经纬度)` 的确定性 Mock 算法 — 同一档案始终得到相同星盘，太阳星座按真实日期范围计算 |
| 城市坐标 | 内置 30+ 中国城市经纬度字典，支持模糊匹配 |
| 每日运势 | 基于用户本命盘与当日真实天象的相位计算（`_analyze_daily_aspects`），六维分数（爱情/财富/事业/学习/人际/综合）从基础分75出发，叠加行星相位强度×权重，clamp 在 50-99 之间；幸运颜色/数字/方位使用 `hash(日期+太阳星座)` 确定性生成，同档案同日结果固定；flatlib 不可用时降级为 MD5 种子的伪随机方案 |
| 星盘生成 | 计算行星位置、ASC/MC、相位关系，支持等宫制，可精确检测行星逆行状态 |

## 四、星盘实现技术

### 4.1 天文计算

| 技术方案 | 说明 |
|------|------|
| **flatlib 库** | 基于 Swiss Ephemeris 的专业占星学计算库，精确计算行星位置、宫位、上升点，支持逆行检测 |
| **计算流程** | 基于出生日期、时间和地理位置，计算太阳、月亮等行星在黄道十二宫中的位置 |
| **宫位体系** | 等宫制（Equal House System），每宫30度，上升点为第一宫的起点 |
| **关键点计算** | ASC（上升点）、MC（天顶）等关键点的计算，用于确定星盘的整体方向 |
| **时区处理** | 使用 pytz 库将用户输入的本地时间转换为计算所需的 UTC 时间 |
| **降级策略** | 如果 flatlib 库不可用，降级使用基于 MD5 哈希的 Mock 方案生成确定性星盘数据 |
| **合盘分析** | 计算两个星盘之间的行星相位关系，分析兼容性分数，支持多维度关系解读 |

### 4.2 星盘前端可视化

| 技术 | 说明 |

| **行星定位** | 根据黄道经度计算 SVG 角度，将天秤座放在12点位置，顺时针排列星座 |

## 六、关键技术决策说明

### 为什么选 SQLModel 而不是 SQLAlchemy + Pydantic 分别用？

SQLModel 统一了 ORM 模型和 API Schema 层，同一个类既是数据库表定义，也是请求/响应的验证模型，减少大量重复代码。代价是版本兼容性较敏感（与 Pydantic v2 的字段命名规则有坑，如字段名不能与类型同名）。

### 为什么前端不用 React/Vue？

当前功能规模（8 页面，逻辑相对独立）不需要组件化框架的复杂度。原生 JS + CSS Variables 已能实现双主题切换、动画、模态框等所有需求，且零编译、零依赖，部署即静态文件。当页面交互复杂度进一步上升（如实时协作、复杂状态管理）时，可迁移至 Vue 3（与现有原生 JS 风格接近）。

### 为什么选择 flatlib 作为占星计算库？

flatlib 是专门为占星学设计的 Python 库，基于著名的 Swiss Ephemeris 计算引擎。相比通用的天文计算库，flatlib 提供了更适合占星学的 API 设计：直接支持星座、宫位、相位等占星学概念，内置逆行检测功能，并且计算精度达到专业占星师的要求。虽然安装可能需要一些编译工具，但现在大多数系统都已具备，而精确的计算结果值得这点额外的配置成本。

### 为什么选 LanceDB 而不是 ChromaDB？

ChromaDB 0.4+ 要求 sqlite3 >= 3.35.0，而 Python 3.8 内置的 sqlite3 是 3.28.0，在 Windows 上无法直接升级。LanceDB 基于 Apache Arrow 格式，无 sqlite3 依赖，兼容 Python 3.8，且支持本地持久化，零配置即可运行。代价是生态相对较小，但对当前规模完全够用。

### 为什么 Embedding 用本地模型而不是 API？

DeepSeek Embedding API 目前对中文支持有限。`paraphrase-multilingual-MiniLM-L12-v2` 专门针对多语言语义相似度优化，支持 50+ 语言包括中文，384 维输出，模型大小约 400MB，本地推理速度快且不消耗 API 额度。首次启动时自动从 hf-mirror.com 下载，后续离线可用。

### 为什么用 SiliconFlow 而不是 OpenAI？

成本考量。SiliconFlow 提供 OpenAI 兼容接口，切换时只需修改 `.env` 中的 `AI_BASE_URL` 和 `AI_API_KEY`，代码零改动。后续如需更换为 DeepSeek、月之暗面（Kimi）等同样兼容的服务，操作相同。

### 为什么 star-chart.html 采用纯 SVG 而不是 Canvas？

SVG 更适合星盘这种静态、结构化的可视化场景：
1. 矢量缩放，任何分辨率下都清晰
2. 元素可单独操作，便于添加交互事件和动态样式
3. 支持 CSS 变量，深色/浅色主题切换更简单
4. 不需要重绘，性能消耗更低
5. 可直接导出为 SVG 文件，便于用户保存和打印

### 为什么合盘分析API使用两个路径？

`/api/synastry` 和 `/api/synastry/calculate` 双路径设计是为了兼容性考虑，两个路径指向同一个处理函数。这样设计允许前端页面使用任一路径进行调用，同时统一了后端实现逻辑，减少了代码重复和潜在的不一致性。对路由做了适当优化，解决了由于路由重复导致的"cannot access results before initialization"等错误。

## 九、安全措施

### 9.1 数据安全

| 措施 | 实现 |
|------|------|
| **密码存储** | bcrypt 单向哈希，不可逆，防彩虹表攻击 |
| **HTTPS 传输** | 全站强制 HTTPS，防中间人攻击 |
| **JWT 过期控制** | 短期访问令牌（2小时）+ 长期刷新令牌（30天）机制 |
| **SQL 注入防护** | 使用 SQLAlchemy ORM 参数化查询，避免字符串拼接 |
| **XSS 防护** | 模板转义、CSP 内容安全策略 |
| **CSRF 防护** | SameSite Cookie + 状态令牌 |
| **错误处理增强** | 合理的异常捕获和变量初始化，防止信息泄露和系统异常 |

### 9.2 API 安全

| 措施 | 实现 |
|------|------|
| **接口限流** | 基于 IP 和用户 ID 的请求频率限制 |
| **输入验证** | Pydantic 模型校验所有 API 请求参数 |
| **权限控制** | 基于角色的访问控制（RBAC）：普通用户/VIP/管理员 |
| **敏感信息过滤** | 日志和错误响应中自动屏蔽密码、令牌等敏感字段 |

## 十、常见问题与解决方案

### 10.1 开发常见问题

| 问题 | 解决方案 |
|------|----------|
| **flatlib 安装失败** | 1. 确保系统已安装 C 编译器和开发工具包<br>2. Windows: 使用 `pip install flatlib-wheels` 预编译版本<br>3. 最坏情况降级到 Mock 模式运行 |
| **MySQL 连接问题** | 1. 检查连接参数，确认 host/port/用户名/密码正确<br>2. 确保数据库服务已启动<br>3. 尝试使用 MySQL Workbench 等工具直接连接测试 |
| **API 返回 500 错误** | 1. 检查日志，确认具体错误原因<br>2. 常见原因：数据库连接失败、数据验证错误、AI API 调用超时<br>3. 使用 try-except 结构改进错误处理，避免变量未初始化问题 |
| **合盘分析出错** | 1. 确保两个档案的星盘数据都已生成<br>2. 检查异常捕获机制，避免"cannot access results before initialization"<br>3. 优化合并重复路由，确保 API 路径一致 |
| **前端显示问题** | 1. 检查浏览器控制台是否有错误<br>2. 清除浏览器缓存和 localStorage<br>3. 检查 localStorage 中存储的 token 是否有效 |

### 10.2 生产部署注意事项

| 项目 | 措施 |
|------|------|
| **配置管理** | 1. 使用环境变量或 .env 文件管理敏感配置<br>2. 不同环境(开发/测试/生产)使用不同配置文件 |
| **数据库连接池** | 1. 设置合理的最大连接数，避免连接耗尽<br>2. 配置连接超时和重试逻辑 |
| **错误监控** | 1. 实现集中式日志记录<br>2. 关键错误邮件/短信告警<br>3. 修复已知的计算和路由错误 |
| **定期备份** | 1. 数据库每日备份<br>2. 文件系统定期快照 |
| **性能优化** | 1. 使用 Redis 缓存热点数据<br>2. 静态资源 CDN 加速<br>3. 合理配置 uvicorn 工作进程数 |
