我认为是“基于 ReAct 框架的大模型智能体迭代推理与数据融合算法”。但如果从垂直业务领域来看，“双路径合盘匹配算法”同样不可或缺。我之所以认为ReAct 智能体迭代推理算法是最核心的，原因如下：1. 它是实现“个性化解读”这个课题目标的“大脑”传统占星软件只能输出固定模板，而通用大模型又存在严重的“幻觉”（比如算错星体位置）。我实现的 ReAct (Reasoning and Acting) 算法，打破了模型“单向输出”的限制。当用户提问时，这个算法强制大模型进入一个 思考(Thought) -> 行动(Action) -> 观察(Observation) 的多轮循环：规划与决策：模型首先“思考”回答这个问题需要什么。例如用户问“我最近事业如何？”，它会判断需要获取实时天象。工具调用（Action）：它会自主生成 Tool Calls，去触发我后端的 search_web（查 DuckDuckGo）或 query_user_data（查数据库里的个人星盘 JSON）。上下文融合与降维：拿到结果后，算法结合底层的“双轨制记忆”（近期对话历史 + 异步提取的长期偏好实体）和 RAG 向量检索结果，最终生成准确的个性化回复。这个算法真正赋予了系统“占星师”的逻辑思维。2. 它是连接“底层硬核数据”与“前端软性交互”的桥梁如果只有这个 AI 大脑是不够的，它的判断依据来自于我业务层面的核心算法——双路径合盘算法。在合盘逻辑中，我设计了 $M \times N$ 的笛卡尔积比对，计算两人数十颗行星在 360 度黄道上的最短绝对夹角（如三分相、刑相），结合古占星的元素权重（水火土风），再通过限制函数（Clamp）输出精确的 0-100 兼容性分数。总结来说：“双路径合盘及运势算法” 保证了系统的科学严谨性，是不产生幻觉的基石；而 “ReAct 智能体迭代推理算法” 则赋予了系统灵魂，将枯燥的天文学数据（如“金火刑相、偏差3度”）转化为像人类占星师那样连贯、有温度、有记忆的自然语言分析。两者的耦合，才真正点题了我毕业设计的核心。


老师您好！在这个项目中，为了实现从底层星盘推演到上层 AI 交互的完整闭环，我综合运用了多种算法。具体可以分为**业务数据推演**、**高可用降级**、**AI与自然语言处理**，以及**前端图形渲染**四大类，我为您详细梳理一下：

### 一、 业务数据推演算法

**1. 双路径合盘算法（用于人际/情侣关系匹配）**
这是系统的一项核心业务算法，采用两阶段计算模型来评估两人星盘的兼容性：

* **第一阶段（古典元素权重分）**：提取两人星盘主星，映射到水、火、土、风四大元素。判断元素组合是同相增强（如火+火加分）、互补调和（如火+风加分）还是冲突（如火+水减分）。
* **第二阶段（相位角笛卡尔积比对）**：利用 $M \times N$ 的双重循环，比对两人所有重要行星的黄道经度。为了计算真实夹角，我使用了最短路径算法：`diff = abs(lon1 - lon2); if diff > 180: diff = 360 - diff`。随后引入“容许度（Orb）”阈值进行判定，例如夹角在 120°±8° 判定为三分相（良性加分），90°±8° 判定为刑相（恶性减分）。
* 最后，通过映射函数将各维度（如爱情、沟通）得分 Clamp 截断限制在 20-100 分制内。

**2. 运势基础分加权与 Clamp 截断算法**
在每日运势计算中，我会计算“本命盘行星”与“当日天象行星”的夹角。将相位映射为不同强度（如合相+5，对分相-1），再乘以该行星在特定领域的权重（例如太阳对事业的权重为2）。计算公式为：`最终分数 = 基础分(75) + Σ(相位强度 × 领域权重)`，为了防止分数溢出，同样使用了 Clamp 函数 `max(50, min(99, score))` 将结果锁定在 50 到 99 分之间。

### 二、 高可用与降级算法

**3. 确定性哈希伪随机算法（Deterministic Hash）**
当底层的天文历书引擎（flatlib）由于环境或依赖问题不可用时，如果完全使用系统的 `random`，会导致用户每次刷新生成的星盘都不一样，体验极差。
为此，我设计了确定性哈希降级算法。提取用户的固定特征（年份+月份+日期+纬度+经度）作为种子进行 MD5 哈希。转化为整数后，对12星座进行取模运算 `(hash_val + i*7) % 12`（使用质数步长确保均匀分布）。这样既做到了伪随机分配，又保证了**同一用户在同一地点生成的星盘结果永远保持一致**（幂等性）。同理，今日幸运色和数字也是基于 `hash(日期+星座)` 算出的。

### 三、 AI 与大模型检索算法

**4. 向量余弦相似度检索算法（用于 RAG 知识库）**
在处理占星学专业文献检索时，我使用了基于 Sentence-Transformers 的 Embedding 模型，将文本映射到 384 维的稠密向量空间中。当用户提问时，使用余弦相似度（Cosine Similarity）算法在 LanceDB 向量库中计算用户问题向量与文档块向量的角度差，从而精确召回 Top-K 的相关占星知识，解决大模型的“幻觉”问题。

**5. ReAct (Reasoning and Acting) 迭代推理算法**
在 `agent.py` 中，我并没有让大模型单向输出，而是使用了 LangChain 的 ReAct 框架。该算法让大模型遵循 `思考(Thought) -> 动作(Action) -> 观察(Observation)` 的循环。如果它发现缺少实时天象数据，它会自动规划调用 DuckDuckGo 搜索工具；如果缺少用户数据，它会调用查库工具。设定了最大 3 次迭代限制以防止死循环。

### 四、 前端可视化算法

**6. 极坐标到直角坐标的转换算法**
为了不在后端耗费服务器资源去生成静态图片，我让前端利用 SVG 原生动态绘制整个星盘。这里用到了基础的三角函数算法，将黄道经度（0°-360°）的极坐标转换为屏幕上 Canvas/SVG 的直角坐标：

* $x = cx + r \times \cos(\theta)$
* $y = cy + r \times \sin(\theta)$
通过这种算法，可以精准定位星体在圆盘上的位置，并自动计算连线路径。
