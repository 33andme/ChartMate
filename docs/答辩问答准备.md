# 毕业答辩问答准备

> 项目：星座星盘系统（ChartMate）
> 技术栈：FastAPI + SQLModel + MySQL + LangChain + LanceDB

---

## 一、项目整体

**Q1：请介绍一下你的项目，解决了什么问题？**

本项目是一个基于占星学的个性化 AI 助手系统。用户输入出生时间和地点后，系统自动计算星盘（太阳星座、月亮星座、上升星座及各行星落座），并基于星盘数据提供每日运势和 AI 对话服务。同时包含用户社区（发帖、评论、点赞）和好友聊天功能。

核心价值是将传统占星学计算（依赖 flatlib + Swiss Ephemeris 天文算法库）与大语言模型结合，让 AI 以"了解用户星盘"的方式进行个性化回答，而不是泛泛的通用回复。

---

**Q2：为什么选择 FastAPI 而不是 Django 或 Flask？**

三个原因：
1. **性能**：FastAPI 基于 ASGI，原生支持异步，适合 AI 对话这种 I/O 密集型场景（调用外部 LLM API 时不阻塞其他请求）。
2. **类型安全**：FastAPI 与 Pydantic 深度集成，请求/响应结构自动校验，减少运行时错误。
3. **自动文档**：自动生成 `/docs` Swagger UI，方便前后端联调，无需手写接口文档。

Flask 太轻量，异步支持不原生；Django 太重，内置 ORM 与 SQLModel 重复。

---

**Q3：系统架构是怎样的？前后端如何分离？**

- **后端**：FastAPI 提供 RESTful API，所有接口以 `/api/` 为前缀，返回 JSON。
- **前端**：纯静态 HTML + Vanilla JS，通过 `fetch` 调用后端 API，挂载在 FastAPI 的 `/static` 路径下。
- **管理后台**：sqladmin 提供 `/admin` 管理页面，支持用户、文档、运势数据的增删改查。
- **数据库**：MySQL（生产）+ SQLModel ORM，表结构在 `models.py` 中集中定义。
- **AI 层**：LangChain Agent 封装在 `agent.py`，调用硅基流动 OpenAI 兼容接口，与主应用解耦。

---

## 二、技术细节

**Q4：星盘是怎么计算的？精度怎么保证？**

使用 `flatlib` 库调用 Swiss Ephemeris（瑞士星历表）进行天文计算。Swiss Ephemeris 是专业天文软件 Astro.com 的底层引擎，精度达到角秒级别，是占星软件的行业标准。

计算流程（`astro_utils.py`）：
1. 将出生城市名用 `geopy` 解析为经纬度
2. 将出生时间转换为儒略日（Julian Day）
3. 调用 `flatlib.Chart()` 计算所有行星位置
4. 提取太阳、月亮、上升点及 10 颗主要行星的黄道经度和所在星座

当 flatlib 不可用时，系统降级为基于出生日期的确定性哈希算法（Mock 模式），保证功能可用但精度有限。

---

**Q5：RAG 是什么？你为什么要用它？**

RAG（Retrieval-Augmented Generation，检索增强生成）是一种将外部知识库与 LLM 结合的技术。

**为什么用**：LLM 的训练数据有截止日期，也无法包含所有占星学专业知识。通过 RAG，管理员可以上传占星学书籍、文章到知识库，AI 回答时会先检索相关内容再生成，使回答更专业、有据可查。

**实现方式**：
1. 文档上传后用 `RecursiveCharacterTextSplitter` 切成 500 字的块
2. 用 HuggingFace 的 `paraphrase-multilingual-MiniLM-L12-v2` 模型将文本转为向量
3. 存入 LanceDB 向量数据库
4. 用户提问时，将问题也转为向量，做余弦相似度检索，取最相关的 3 条注入 System Prompt

---

**Q6：AI 记忆功能是怎么实现的？**

每次对话结束后，系统在后台异步（`asyncio.create_task`）调用一个轻量 AI 模型（Qwen3-8B）分析最近 6 条对话，提取值得长期记住的信息，结构化为 JSON：

```json
[{"type": "entity", "content": "用户男友是双子座"},
 {"type": "preference", "content": "用户不喜欢太长的回答"}]
```

去重后写入 `user_memories` 表。下次对话时，这些记忆会追加到 System Prompt，让 AI 像"认识用户"一样回答。

去重策略：对比新记忆与已有记忆的前 20 个字，相同则跳过，避免语义重复的记忆堆积。

---

**Q7：JWT 认证是如何工作的？Token 过期了怎么办？**

用户登录后，服务端生成 JWT（JSON Web Token），包含用户 ID、邮箱和过期时间，用 HS256 算法和密钥签名后返回给前端。

前端将 Token 存入 `localStorage`，后续所有需要认证的请求在 Header 中带上 `Authorization: Bearer <token>`。

服务端通过 FastAPI 的 `Depends(get_current_user)` 依赖注入机制，在每个需要认证的接口中自动解码验证 Token。

**Token 过期**：当前有效期设为 7 天，过期后返回 401，前端跳转到登录页重新登录。项目未实现 Refresh Token，这是一个可以优化的点——可以加一个 `/api/auth/refresh` 接口，用旧 Token 换新 Token（在有效期内）。

---

**Q8：数据库设计中有哪些值得说的设计决策？**

三个亮点：

1. **好友关系即聊天室**：没有独立的好友表，`chat_rooms` 表本身就代表好友关系。`user1_id` 强制为较小的 ID，`user2_id` 为较大的 ID，加上 UniqueConstraint，保证同一对用户只有一条记录，无论谁发起都能命中同一行。

2. **星盘数据 JSON 化存储**：`profiles.astral_config` 字段存储完整星盘的 JSON 字符串，而不是拆成几十个字段。因为星盘数据结构复杂（行星 × 属性），JSON 存储更灵活，查询时整体读取再 Python 层解析。

3. **运势缓存**：每日运势计算涉及天文计算，同一档案同一天结果相同，存入 `daily_fortunes` 表缓存，当天再次请求直接返回，避免重复计算。

## 三、难点与问题

**Q9：开发过程中遇到了哪些难点？怎么解决的？**

遇到了三个典型问题：

**1. Swiss Ephemeris 路径问题**
`flatlib` 内部会在每次 `Chart()` 实例化时重置星历文件路径到默认的 `\sweph\ephe\`，导致找不到 `.se1` 星历文件。
解决：在每次调用 `Chart()` 之前，手动调用 `swisseph.set_ephe_path()` 将路径设回 flatlib 资源目录，确保路径生效。

**2. bcrypt 版本兼容问题**
`passlib 1.7.x` 会访问 `bcrypt.__about__.__version__`，但 `bcrypt 4.x` 删除了 `__about__` 属性，导致 `AttributeError`。
解决：在 `auth.py` 中加了一个兼容补丁，检测 `__about__` 不存在时手动创建，无需降级依赖版本。

**3. 管理员删除用户 500 错误**
MySQL 外键约束要求先删子表再删父表，原来的删除逻辑漏掉了 `chat_messages` 和 `user_memories`，触发外键约束报错。
解决：整理出完整的删除顺序：chat_messages → chat_rooms → user_memories → 其他关联数据 → user，按序删除。

**Q10：系统有哪些安全隐患？如何改进？**

诚实地说有几点不足：

1. **SECRET_KEY 硬编码**：JWT 签名密钥在代码里有默认值，生产部署必须通过环境变量替换，否则任何人都能伪造 Token。
2. **CORS 全开放**：`allow_origins=["*"]` 适合开发，生产应限制为具体域名。
3. **文件上传无病毒扫描**：上传接口只检查了文件扩展名和大小，实际内容未做扫描。
4. **无速率限制**：发送验证码接口可被暴力刷，应加 IP 频率限制（如 `slowapi`）。
5. **进程内 AI 记忆缓存**：多实例部署时各进程缓存不同步，应换用 Redis。

**Q12：为什么选择 LanceDB 而不是 Pinecone 或 Milvus？**

LanceDB 是本地文件型向量数据库，无需单独部署服务，直接写入本地目录即可。对于本项目这种单机部署的原型系统，LanceDB 的零运维成本是最大优势。

如果未来用户量增大、需要多机部署，可以迁移到 Milvus（开源，支持分布式）或 Pinecone（云托管）。LangChain 的 VectorStore 接口做了抽象，切换时只需改初始化代码，业务逻辑不变。

## 五、代码与算法细节

**Q13：黄道经度是怎么换算成星座的？代码怎么写的？**

黄道是一个 360° 的圆，12 个星座每个占 30°，白羊座从 0° 开始。换算逻辑在 `astro_utils.py` 的 `_lon_to_sign_info()` 函数：

```python
def _lon_to_sign_info(lon_deg: float) -> dict:
    lon_deg = lon_deg % 360          # 防止超出 360
    sign = SIGNS_ORDER[int(lon_deg / 30)]   # 整除 30 得到星座索引
    return {
        "sign":   sign,
        "degree": round(lon_deg % 30, 2),   # 在该星座内的度数
    }
```

例如木星黄道经度 95°：`95 / 30 = 3`，对应 `SIGNS_ORDER[3]` 即巨蟹座，在巨蟹座内 `95 % 30 = 5°`。

---

**Q14：相位角度是怎么计算的？为什么要用 `360 - diff`？**

相位（Aspect）是两颗行星黄道经度之差的绝对值。但黄道是圆形，两点之间有"近路"和"远路"，取较小的那个角才是真实夹角。代码在 `_get_aspect_angle()`：

```python
def _get_aspect_angle(lon1: float, lon2: float):
    diff = abs(lon1 - lon2)
    if diff > 180:
        diff = 360 - diff   # 取劣弧（较小角）
    return diff
```

例如：行星 A 在 10°，行星 B 在 350°，直接相减得 340°，但绕另一方向只有 360-340=20°，所以真实相位是 20°（接近合相 0°），而不是 340°。

**Q15：运势分数是怎么算出来的？有没有公式？**

运势计算分两层（`astro_utils.py` 的 `_calculate_fortune_scores` + `_analyze_daily_aspects`）：

**第一层：相位强度**
比较用户本命盘各行星与当日天象各行星的相位角，映射成强度值：

| 相位 | 角度 | 容许度 | 强度 |
|------|------|--------|------|
| 合相 | 0°  | ±8°   | +5  |
| 三分相 | 120° | ±8° | +4 |
| 六分相 | 60° | ±6°  | +3 |
| 对分相 | 180° | ±8° | -1 |
| 刑相 | 90° | ±8°   | -2 |

无相位则强度为 0。

**第二层：行星权重叠加**
不同行星影响不同维度，强度乘以权重累加到基础分 75：

```python
planet_weights = {
    "Sun":     {"overall": 3, "career": 2, "social": 1},
    "Venus":   {"love": 3, "social": 2, "wealth": 1},
    "Mercury": {"study": 3, "career": 1, "social": 1},
    "Jupiter": {"wealth": 3, "overall": 2, "career": 1},
    ...
}
# 最终分数 = max(50, min(99, 75 + Σ(强度 × 权重)))
```

例如：当日太阳与本命太阳形成三分相（强度 +4），太阳对 overall 权重为 3，则 overall 加分 +12，从 75 变成 87。

**Q16：Mock 模式下星盘数据是怎么生成的？为什么不是随机的？**

Mock 模式需要满足一个关键要求：**同一个用户每次得到完全相同的星盘**，否则用户每次刷新星盘都不一样，体验极差。

实现方式是用 MD5 哈希生成确定性种子（`_calculate_mock()`）：

```python
seed_str = f"{birth_time.year}{birth_time.month}{birth_time.day}{lat}{lon}"
hash_val = int(hashlib.md5(seed_str.encode()).hexdigest(), 16)

# 每颗行星用不同偏移量，确保12颗行星落座各不相同
for i, pid in enumerate(planet_ids):
    sign = SIGNS_ORDER[(hash_val + i * 7) % 12]
```

`hash_val + i * 7` 让每颗行星的偏移量不同（步长 7 是质数，与 12 互质，保证分布均匀），太阳星座最后用真实日期推算覆盖，保证准确性。

**Q17：合盘（Synastry）兼容性分数是怎么计算的？**

合盘分析两个人星盘之间的相位关系，不同行星组合影响不同的感情维度（`_calculate_compatibility_scores()`）：

```python
planet_influences = {
    ("Venus", "Mars"):   {"love": 4, "sexual": 5},   # 金火相位：爱情和吸引力
    ("Sun", "Moon"):     {"overall": 4, "emotional": 3},  # 日月相位：深层共鸣
    ("Mercury","Mercury"):{"communication": 4},        # 双水相位：沟通契合
    ("Moon", "Moon"):    {"emotional": 4, "overall": 2},
}
```

相位强度（正相位 +，负相位 -）× 权重，叠加到各维度基础分 60 上，最终限制在 20-100 分之间。

例如甲的金星与乙的火星形成三分相（强度 +4），则 `love += 4×4=16`，`sexual += 4×5=20`，两人爱情和性吸引力分数大幅提升。

**Q18：等宫制是什么？你的宫位是怎么计算的？**

占星中有多种宫位制，本项目使用最简单的**等宫制**：以上升点（Asc）为第一宫起点，每宫固定 30°，沿黄道依次排列 12 宫（`_calculate_with_flatlib()` 第 300 行）：

```python
asc_deg = asc.lon   # 上升点黄道经度
for i in range(1, 13):
    h_deg = (asc_deg + (i - 1) * 30) % 360
```

等宫制的优点是计算简单、在高纬度地区不会出现极端变形；缺点是不如普拉西德宫位制精确（天文学上各宫弧度并不等分）。这是一个可以升级的点——换用 Placidus 宫位制只需修改 flatlib 的 `hsys` 参数。

**Q19：`_fallback_fortune_scores` 里为什么用 `random.Random(hash_val)` 而不是直接 `random`？**

直接用全局 `random` 会受到程序运行状态影响，同一天同一用户两次请求结果可能不同（如果中间有其他代码调用了 random）。

```python
seed = f"{target_date}{sun_sign}{moon_sign}"
hash_val = int(hashlib.md5(seed.encode()).hexdigest(), 16)
rng = random.Random(hash_val)   # 独立的 Random 实例，不影响全局状态
return {
    "overall": rng.randint(65, 95),
    ...
}
```

`random.Random(hash_val)` 创建一个**独立的随机数生成器实例**，用固定种子初始化，与全局 `random` 互不干扰，保证同一天同一用户的运势分数永远相同，符合"当日运势"的语义。

**Q20：为什么城市坐标用硬编码字典，不用 API 实时查询？**

有三个考虑：

1. **离线可用**：部署环境可能没有外网，硬编码字典无网络依赖
2. **响应速度**：字典查找是 O(1)，geopy API 需要网络往返，星盘计算本身已有耗时
3. **足够覆盖**：项目面向国内用户，字典覆盖了国内约 80 个主要城市和别名（如"北平"→北京）

模糊匹配作为补充：如果用户输入"南京市"而字典只有"南京"，通过 `city in city_name or city_name in city` 的子串匹配也能命中（`get_city_coords()` 第 138 行）。找不到时 fallback 到北京坐标并在返回数据中附带 `warning` 字段提示用户。
